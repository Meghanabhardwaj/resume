# Resume Repo — Notes for AI Agents

This repo is the source of truth for `Aditya_SWE_Resume_2YOE.pdf`. The PDF lives on the GitHub Releases page and the portfolio site (`adysfolio.vercel.app`) renders it via that URL.

## Where to edit

- **`resume.yaml`** — the only source of truth for content. Edit here, never edit `sections/*.tex` directly. Those are generated by `scripts/generate_resume.py`.
- **`formatting.sty`** — typography, spacing, and layout primitives.
- **`.claude/skills/resume-bullets/SKILL.md`** — the house style for bullets. Read it before adding or rewriting any bullet.

## How a change reaches the website

```
push to main
  → GitHub Actions (.github/workflows/release.yml)
      ├── runs scripts/generate_resume.py
      ├── builds PDF inside the latex-builder Docker image
      ├── creates a Release tagged v<run_number> with the PDF attached
      └── POSTs to the portfolio's /api/revalidate-resume endpoint
              ↓
          portfolio invalidates its Next.js fetch cache and Vercel
          edge cache, so the next visitor sees the new PDF immediately
```

The path filter on the workflow only triggers on `**.tex`, `**.cls`, `**.sty`, `**.json`, `**.yaml`, `scripts/**`, and `.docker/**`. Changes to `.claude/`, `README.md`, etc. won't rebuild — that's fine, those don't affect the PDF.

## When to manually revalidate

The release workflow pings the portfolio automatically. You should only need the manual curl when:

- The CI run failed or was cancelled but the GitHub Release was somehow created another way.
- Someone uploaded a release artifact by hand.
- You're testing the revalidation endpoint itself.

```bash
curl -fsS -X POST "https://adysfolio.vercel.app/api/revalidate-resume" \
  -H "Authorization: Bearer $PORTFOLIO_REVALIDATE_SECRET" \
  -H "Content-Type: application/json" \
  -d '{"source":"manual"}'
# → {"ok":true,"revalidated":["resume"],"at":...}
```

`PORTFOLIO_REVALIDATE_SECRET` is the same value the GitHub Actions workflow uses; ask the repo owner or pull it from `gh secret list` if you have access.

## CI configuration the revalidation step depends on

For the workflow's revalidation step to fire, the resume repo must have:

- `vars.PORTFOLIO_REVALIDATE_URL` — repository variable, e.g. `https://adysfolio.vercel.app/api/revalidate-resume`
- `secrets.PORTFOLIO_REVALIDATE_SECRET` — repository secret, must match `RESUME_REVALIDATE_SECRET` configured on the portfolio's Vercel project

If `PORTFOLIO_REVALIDATE_URL` is empty, the step is skipped silently (the `if:` guard) — so missing config never breaks the build, it just means the cache won't auto-bust.

## Standard workflow when editing the resume

1. Edit `resume.yaml` (or `formatting.sty` if it's a layout fix).
2. Run `python3 scripts/generate_resume.py` to regenerate `sections/*.tex`, `resume.json`, `schema.json`.
3. `make compile` to verify the PDF renders correctly. Hard limit: every project/role bullet fits on one rendered line.
4. Commit in logical units (content, layout fixes, spacing tweaks each their own commit) and push to `main`.
5. CI takes ~2–4 minutes. Confirm a new release appears at https://github.com/adityaongit/resume/releases/latest and that the portfolio is serving the new PDF.

If the portfolio still shows the old PDF more than a minute after CI completes, run the manual curl above before assuming there's a deeper issue.
